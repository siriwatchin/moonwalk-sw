Moon Walk · Gait-Sensing Layer

Sensor → Six Gait Metrics

How a sensor box clipped to a walking cane turns raw IMU signals into gait metrics — and which are measured as absolutes versus per-user trends. Measure-and-trend, not diagnostic (ADR-0001).

Current build = IMU-only. The only sensor physically in hand is the Arduino Nano 33 BLE's onboard LSM9DS1 IMU. The designed multi-FSR Handle Load grip and the ToF tip sensor (CONTEXT.md / ADR-0002) are not yet acquired. All six metrics are derivable from the IMU alone; the FSR & ToF are optional upgrades that tighten the loading and spatial metrics.

1The sensing principle — one Stick Cycle, IMU only

The cane pivots about its tip. One Stick Cycle (plant → load → push → swing) runs in phase with the user's gait — a proxy for it, not identical. With no force sensor, the IMU alone separates the planted phase (stick stationary) from the swing phase (stick rotating).

Plant

PHASE 2
Load
IMU: stick near-vertical, low angular rate (inferred "planted")
FSR (future): would measure load magnitude directly
PHASE 3
Push
IMU: acceleration along shaft begins; angular rate rising
FSR (future): load decaying
PHASE 4
Swing
IMU: gyro integrates swing arc θ; high angular rate
FSR (future): load ≈ 0

IMU does everything

6-axis accel+gyro (magnetometer ignored). The stationary tip-plant is detected from low angular rate + an impact spike — that same instant is the zero-velocity (ZUPT) reset. Gyro integration over the swing gives θ.

No force = motion proxy

Without an FSR, "planted/loaded" is inferred from the stick being still, not from measured force. This is fuzzier at plant/lift edges, and a light touch that bears no weight still reads as "planted".

Stick ≠ leg

We instrument the stick, not the leg, and there is no force plate. Temporal metrics are honest Stick-proxies; spatial metrics are trend-only.

4Calibration & per-user Baseline

Mode & stick length L

Cane vs Walker is set manually at setup (ADR-0002) and selects the Distance Estimator's model. Stick length L is the geometric scale for every spatial metric — L·sin(θ) is meaningless without it.

FSR grip threshold — N/A in this build

The designed per-user FSR load threshold (PRD US-4) doesn't apply yet — there's no FSR. Its stand-in is the IMU motion/stillness threshold that decides "planted vs swinging". Re-enable the load calibration when an FSR is added.

Continuous Baseline → Drift

The device learns this user's own normal on-device. Only sustained Drift raises an Alert; a single odd walk must not. Because trends are user-vs-self, any constant stride underestimation cancels out for change-detection.

5Hardware reality check

✅ IMU covers all six

LSM9DS1 ranges (±2000 dps / ±16 g, up to 952 Hz) comfortably exceed cane motion. 100–200 Hz resolves plant impacts and the swing arc. No second sensor is required to produce any metric.

⚠️ FSR & ToF not in hand

The docs design in a multi-FSR Handle Load grip and a ToF tip sensor; neither is acquired. Effect: stance time & duty factor are motion-inferred, not force-measured; and the offline Proximity Alert can't run without the ToF.

🔌 Constraints & upgrades

3.3 V only — any future FSR drives off 3.3 V to one analog pin (~$5–15). No onboard charger; no microphone (plain Nano 33 BLE, not the Sense). UART protocol & BLE GATT are hand-rolled.
