Moon Walk · Flagship Application

Weight Support Feedback Cane — Architecture

Moon Walk is one clip-on sensor running several applications (ADR-0009). The flagship WSFC runs a real-time weight-support biofeedback loop that retrains a rehab Patient to load the affected leg instead of over-leaning on the cane. Wellness gait-monitoring and the Speaking Stick are secondary apps on the same hardware.

Hardware reality (2026-05-27). In hand & working: the Arduino Nano 33 BLE's onboard LSM9DS1 IMU and LPS22HB barometer, plus the pneumatic Handle Load bladder — now built, bench-calibrated, and drift/hysteresis-validated (ADR-0010). Not acquired: the ToF Distance tip sensor. So the whole WSFC loading loop is now unblocked; only the ToF-dependent Proximity Alert remains gated.

1Two boards, one sensor (ADR-0004)

The over-lean cue must be immediate and offline, so the WSFC Feedback Loop runs on the always-on Nano; the Threshold Engine (which only updates the target band per session/week) runs on the UNO Q's Linux side with history and the wellness intelligence. Health data never leaves device + phone.

2The WSFC real-time loop — rule-based + DSP (ADR-0011)

The flagship loop is rule-based + DSP, not ML. It decomposes into three on-device modules plus the per-session Threshold Engine. ML is a deferred, named fallback for segmentation only — never in the cue decision, and never classifying "abnormal" / "fall-risk".

Nano · per step, real-time

① segment → ② condition load → ③ decide cue

①Swing/Step Segmenter
RULE-BASED — gyro threshold + refractory (extends CycleDetector). Marks step boundaries + the unloaded-swing window.
②Load Conditioner
DSP — session-start tare → 2nd-order polynomial calibration → relative load (% of baseline) · EMA smoothing.
③Cue Decider
THRESHOLD — peak load per step vs target band · Schmitt deadband + min on/off → beep / haptic.
UNO Q · per session / per week

④ set & fade the target band

④Threshold Engine
baseline = percentile of per-step-peak load distribution (setup walk) · target faded −10%/week (≈60%→30%) · advance only when ≥80% of a week's steps land in-band

🔁 Tare strategy (ADR-0010, amended)

WSFC tares the barometer once at session start, after a brief grip/thermal equilibration, with a clinician one-button manual re-zero — not per-swing auto-tare (that is rescoped to the long, unsupervised wellness app). The Segmenter's swing detection is used for step segmentation only, decoupled from the load zero, so a missed swing can't inject a bad re-zero into a safety cue. Because the target is a % of a baseline measured in the same session after equilibration, slow common-mode drift largely cancels in the ratio.

⛔ Claim-safety binds the loop (ADR-0009)

The target is always % of the Patient's own baseline cane-dependence — never %-body-weight, never an absolute-force claim; kgf is bench-calibration / optional clinician readout only. No diagnosis, no fall-risk. ML is foreclosed from exactly the "abnormality / fall-risk" outputs the boundary bans.

📉 Honest evidence framing

The pressure-sensor + auditory-cue class did not reach significance for gait speed in the Wang 2025 meta-analysis (SMD 0.30, P=0.05); supportive effects are on loading & muscle-activation (Jung, Kang, Tamburella). WSFC's pitch is generating our own evidence — see rehab/wsfc-processing-references.md.

4One Stick Cycle — what the sensors see

One Stick Cycle (plant → load → push → swing) runs in phase with gait — a proxy, not identical to it. The LOAD peak is where the Cue Decider makes its once-per-step decision; the SWING phase (cane in air, load ≈ 0) is the natural load-zero reference.

PHASE 1
Plant
IMU: angular rate → 0 + accel impact → plant event & ZUPT reset
Load: rises as weight transfers to the cane
PHASE 2
Load
IMU: stick near-vertical, low angular rate
Load: peak — Cue Decider samples here (once per step)
PHASE 3
Push
IMU: acceleration along shaft; angular rate rising
Load: decaying
PHASE 4
Swing
IMU: gyro integrates swing arc θ; high angular rate
Load ≈ 0: the load-zero reference (wellness re-zeros here)

5Secondary applications (same hardware)

🩺 Wellness gait monitoring

The UNO Q learns the User's Baseline and scores sustained Drift to raise a non-medical Alert (single anomaly ≠ Alert). Measure-and-trend only (ADR-0001/0005); real-time cues only if Training Mode is opted in (ADR-0006). Per-swing auto-tare lives here.

🗣️ Speaking Stick

Orthogonal see-and-speak layer (ADR-0003): camera → cloud VLM → TTS, plus an offline Proximity Alert (gated on the ToF). Not part of the WSFC clinical story.

🚫 What we cannot measure

No leg stance time (the handle is loaded, not the foot), no ground reaction force, no absolute force / Newtons, no %-body-weight, no diagnosis, no fall-risk. Binds all applications (FEATURES.md, CONTEXT.md Claim Safety).
